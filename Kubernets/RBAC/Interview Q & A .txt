Kubernetes RBAC – Senior Interview Q&A (Real-Time)
✅ 1. What is RBAC in Kubernetes?

Answer (Senior Level):
RBAC (Role-Based Access Control) in Kubernetes is used to control who can access what resources and perform which actions within a cluster.

It works using:

Role / ClusterRole → Defines permissions
RoleBinding / ClusterRoleBinding → Assigns permissions to users/groups/service accounts

👉 Real-time:

Used to restrict developers to only specific namespaces, avoid cluster-wide access.

✅ 2. Difference: Role vs ClusterRole
Feature	Role	ClusterRole
Scope	Namespace	Cluster-wide
Use case	App-level access	Admin / shared resources

👉 Example:

Role → Dev team access only dev namespace
ClusterRole → Monitoring tools (Prometheus) access all namespaces
✅ 3. RoleBinding vs ClusterRoleBinding
Feature	RoleBinding	ClusterRoleBinding
Scope	Namespace	Entire cluster
Binds	Role/ClusterRole	ClusterRole only

👉 Real-time:

RoleBinding → Give access to one namespace
ClusterRoleBinding → Give admin access to entire cluster
✅ 4. How do you give read-only access to a namespace?

Answer:

apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: dev
  name: read-only
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "watch", "list"]
kind: RoleBinding
metadata:
  name: read-binding
  namespace: dev
subjects:
- kind: User
  name: dev-user
roleRef:
  kind: Role
  name: read-only
  apiGroup: rbac.authorization.k8s.io

👉 Real-time:

Developer can only view pods, not create/delete.

✅ 5. What are verbs in RBAC?

Common verbs:

get
list
watch
create
delete
update
patch

👉 Senior tip:

Never give * (all verbs) in production unless required.

✅ 6. How RBAC works internally?

👉 Flow:

User sends request → kubectl
API Server authenticates user
RBAC checks:
Role / ClusterRole rules
Bindings
Allow or deny

👉 Real-time debugging:

kubectl auth can-i create pods --as dev-user -n dev
✅ 7. How do you troubleshoot RBAC issues?

Steps:

Check error:

forbidden: User cannot create pods

Verify permissions:

kubectl auth can-i create pods --as dev-user -n dev

Check roles:

kubectl get role,rolebinding -n dev

Describe:

kubectl describe role read-only -n dev

👉 Real-time:

Most production issues = missing RoleBinding or wrong namespace.

✅ 8. ServiceAccount vs User in RBAC
Feature	User	ServiceAccount
Used by	Humans	Pods
Authentication	External	Kubernetes

👉 Real-time:

Pods use ServiceAccount to access API
Example: Jenkins, CI/CD pipelines
✅ 9. How to give Pod access to Kubernetes API?

👉 Use ServiceAccount:

apiVersion: v1
kind: ServiceAccount
metadata:
  name: app-sa

Bind it:

kind: RoleBinding
subjects:
- kind: ServiceAccount
  name: app-sa

👉 Real-time:

Used in CI/CD pipelines, operators, controllers

✅ 10. What is least privilege principle?

👉 Only give minimum required permissions

❌ Bad:

verbs: ["*"]
resources: ["*"]

✅ Good:

verbs: ["get", "list"]
resources: ["pods"]

👉 Senior answer:

Prevents security breaches and accidental deletions.

✅ 11. Can Role access multiple namespaces?

❌ No
👉 Role is namespace-specific

✅ Use ClusterRole + RoleBinding for multi-namespace access

✅ 12. Real-Time Scenario Question
❓ Dev team cannot deploy pods in dev namespace. What do you do?

👉 Answer:

Check error

Verify permissions:

kubectl auth can-i create pods --as dev-user -n dev
Check RoleBinding exists
Ensure correct namespace
Fix YAML if needed

👉 Final:

Issue mostly due to missing create verb or wrong binding

✅ 13. Difference: apiGroups
apiGroups: [""]

→ Core resources (pods, services)

apiGroups: ["apps"]

→ deployments, replicasets

👉 Interview trick question 🔥

✅ 14. How do you give full admin access?
kind: ClusterRoleBinding
roleRef:
  name: cluster-admin

👉 Real-time:

Used only for cluster admins, not developers

✅ 15. What is Aggregated ClusterRole?

👉 Advanced concept:

Combine multiple ClusterRoles
Used in large organizations

Default roles? → cluster-admin, admin, edit, view
How to secure cluster? → RBAC + Network Policies + IAM
Can we restrict secrets? → Yes via RBAC rules

🔥 Section 1: Concept Traps
1. Can a Role grant access to multiple namespaces?

👉 ❌ No
✔️ Only works in one namespace

2. Can a RoleBinding reference a ClusterRole?

👉 ✅ Yes (very common trick)

3. Can a ClusterRole be namespace-scoped?

👉 ❌ No
But 👉 can be used inside namespace via RoleBinding

4. What happens if no RoleBinding exists?

👉 ❌ Access denied (default deny)

5. Which is more secure: Role or ClusterRole?

👉 ✅ Role (limited scope)

⚠️ Section 2: Real-Time Scenario Questions

👉 Missing delete verb

👉 Missing binding (ClusterRoleBinding)

👉 ✅ Yes (best practice)
